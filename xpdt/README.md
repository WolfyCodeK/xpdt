# xpdt config (customised xplr)

A heavily customised [xplr](https://xplr.dev) (v1.1.0) file manager set up as a lightweight, keyboard driven git client and code browser. It adds a git author column, a git status column, a live changes browser (stage / unstage whole files or individual hunks / discard / commit / edit in Neovim), a commit history browser with undo, a stash browser (create / apply / pop / drop / clear), an inline full file diff viewer, recursive file and content search with a persistent scope toggle, a syntax highlighted file preview with indent aware wrapping and clipboard copy, file create / rename / delete / move, a per-action confirmation gate (type two random digits before an action runs, on by default for every mutating action and toggleable in a settings menu), selectable colour themes (Monokai, Gruvbox, Nord, Dracula, Tokyo Night) that recolour xpdt, Neovim, bat and fzf together, opt-in per-language Neovim intellisense you turn on from the same settings menu, Nerd Font icons and colours, and on-demand help popups (`h` for the controls, `ctrl-h` for a neovim cheat sheet).

Everything lives in `~/.config/xpdt/`. `init.lua` is the entry point; the rest are small helper scripts it shells out to. It is loaded by the `xpdt` command (`xplr -c ~/.config/xpdt/init.lua`); plain `xplr` is left untouched and runs stock.

## Layout

`init.lua` replaces the default layout with a `Dynamic` custom layout (`custom.render_layout`) stacked vertically:

1. `Table` (the file listing) with custom columns: index, path (tree + icon + name), a one char git modified dot (`M`), git author, size, modified time.
2. `changes` box (`custom.render_git_changes`) listing staged and unstaged changes, auto sized to the number of changes (capped at 30 rows).
3. `git history` box (`custom.render_git_graph`) showing the last 100 commits of the current branch. Each commit is prefixed with a filled dot (`●`) if it is on the upstream (pushed) or a hollow yellow dot (`○`) if it is local - ahead of the upstream and not yet pushed (with no upstream, every commit keeps the plain dot). Sized responsively from the terminal height: on a short window it scales down first (to a ~3 row floor) so the file listing keeps at least ~10 rows; only once the history reaches that floor does the file listing itself start to shrink.
4. `claude` box (`custom.render_claude`), only present when the `claude-integration` setting is on and you have recent Claude Code sessions (otherwise 0 height / hidden). A grouped, cross-repo panel of your sessions with status, context % and last-active time. See the claude panel below.
5. `InputAndLogs` (xplr's built in input / log line).

The key legend is no longer an always-on panel (it ran off the side of narrow terminals). It is now an on-demand popup: `h` for the xpdt controls, `ctrl-h` for the neovim cheat sheet.

Every mutating action (create / move / delete, stage / hunk / discard / commit, stash apply / pop / drop / new / clear, undo commit, git checkout / pull) runs behind the confirmation gate: by default it asks you to type two random digits first. The `,` menu turns that off globally or per action. See Confirmation gate below.

Rename and move (`m` / `M`): `m` renames the focused item with its current name **pre-filled to edit** (no retyping); `M` moves it into a folder you pick by **fuzzy-filtering** the tree from the launch dir (no path typing). Both run behind the confirmation gate's `move` action.

Changes browser (`enter`): `s` stage/unstage the whole file (toggle, based on where the entry currently is), `p` open the hunk browser to stage/unstage individual hunks of the focused file, `d` discard, `c` commit (prompts for a message), `r` refresh the list (pick up changes made outside xpdt while you sit in the browser), `→` edit the file in Neovim if the entry is **unstaged**, or open the read-only inline diff if it is **staged** (you cannot edit a staged snapshot - unstage it with `s` first if you want to edit it), `←` back. `enter` does nothing here (opening a file is `→` only, app-wide). Stage / hunk / discard / commit go through the confirmation gate. The list is sized to the number of changes (capped at 20, then it scrolls), with the rest of the screen given to the diff preview. It opens even when there are no changes (you land on an empty list under the same header), so you can sit in it and press `r` to watch changes appear.

Hunk browser (`p`): a `git add -p` style view of the focused file's hunks. `s` stages the focused hunk (or unstages it, if you opened `p` on a staged entry); `d` discards the hunk (unstaged hunks only - it reverts that hunk in the working tree); `←` back. Each hunk shows its diff in the preview; acting on a hunk reloads the list so you can work through them one at a time. Both `s` and `d` go through the confirmation gate (`hunk` and `discard` respectively).

Commit history (`;`): `→` open a commit, `b` view a different branch's / ref's commits (local or remote), `ctrl-p` cherry-pick the focused commit onto the current branch, `ctrl-z` undo the last commit (soft reset), `←` back. Inside a commit: `→` inline diff viewer, `←` back. Typing filters the commits but keeps them **newest-first** (the list is `--no-sort`, so it stays in git-log / date order instead of fzf re-ordering matches by relevance). Each commit is tagged with a local-vs-pushed dot (yellow ○ = local / not pushed, ● = pushed), the same marking as the history box. The header shows which branch you are viewing and which branch a cherry-pick would land on.

Cherry-pick (`ctrl-p` in the history browser): applies the focused commit onto your current branch. The working tree must be clean first (git requires it), it goes through the confirmation gate, and on a conflict it auto-aborts so the branch is left exactly as it was. Point the browser at another branch with `b`, then `ctrl-p` the commit you want.

Stash browser (`s`): `a` apply (keeps the stash), `p` pop (apply then drop), `d` drop the selected stash, `n` new stash from the working tree (prompts for an optional message, includes untracked files), `x` clear all stashes, `→` view the full stash diff in a pager, `←` back. All five actions go through the confirmation gate. The preview shows the stash's diff with word-level highlighting. Actions stay on letter keys and viewing is on `→`; `enter` does nothing, so neither a stray keypress nor viewing can ever mutate a stash.

File / content search (`/` `\`): type to filter, `tab` toggle scope (current dir vs whole tree from the launch dir), `←` delete back through the query (exit the search once it is empty). In `/` (files), `→` confirms the focused hit - opening a file in Neovim, or entering a folder. In `\` (in-files), `→` opens the file on the matched line in Neovim. `enter` is not bound to open anything, in either search: opening a file is `→` only, across the whole app. Hits are shown as paths relative to the scope dir (leading `/`, e.g. `/sub/file`), not the full launch path.

File preview: type to filter lines, `→` (or `ctrl-e`) open in Neovim at the line, `ctrl-y` copy the whole file to the clipboard, `←` back (`enter` does nothing). Clearing the search keeps you on the line it took you to, instead of jumping back to the top.

Inline diff viewer: `→` next change, `shift-→` previous change, `←` back. The focused change is kept centred on screen as you move between changes.

## How the features work

### Git author column (the performance sensitive one)

`fmt_general_table_row_cols_2` fills the author column. Naively this ran one `git log -1` per file per render, which was very slow. Now:

- Results are cached per absolute path for the session (`git_author_cache`), so each entry is computed once.
- On the first cache miss in a directory, `batch_git_authors` calls `git-authors.sh` once for the whole directory (`git_author_dir_done` guards against re running it).
- `git-authors.sh` branches: a leaf directory (no subdirectories) uses one recursive `git log --name-only -- .`; a directory with subdirectories batches only its immediate files (`git ls-tree` + `git log` over those paths) so it never walks the whole repository subtree. This is the fix for the top of the repo taking a couple of seconds to open.
- That batch is bounded to the most recent `-n 500` commits. `git log --name-only` has no way to stop once it has seen each file, so without a bound it walks the repo's entire history every time you enter a directory - the main lag on a deep repo. A file's last author is almost always within that window; older files simply show no author.
- Anything not covered by the batch (an untracked file, or a file whose last commit is older than the window) is left blank - not resolved with a per-file `git log`. That per-file fallback used to spawn a `git` process for every such entry while the table rendered, which was a large part of the first-open lag for a directory full of new/untracked files.
- A gitignored path shows `ignored` in bold red instead of an author. `git_status` runs `git status --porcelain --ignored` (git collapses an ignored directory to a single `dir/` entry, so this stays cheap), and those `!!` rows are kept out of the changes box and the M column - they only feed the author column's ignored flag (`path_ignored`).

### Git modified column and status

`custom.git_modified` shows a coloured dot when a path is dirty. It reads `git_status`, which caches `git status --porcelain --ignored` per repo root. The changes box (`render_git_changes`) and the browser reuse the same porcelain parse.

Git state (status and the history graph) is **repo-wide**, so it is cached across directory navigation rather than recomputed every time you move. Earlier, `on_directory_change` cleared the status cache, so entering any directory re-ran `git status` over the whole worktree - the main lag when exploring a big repo. Now:

- The caches persist across directory changes and expire only on a longer TTL (`STATUS_TTL` / `GIT_LOG_TTL`, ~10s) - a backstop for changes made outside xpdt (e.g. an edit in another terminal), which the panels then catch up to within that window.
- After any xpdt action that can change git state - stage / commit / discard (changes browser), stash, checkout / pull (git menu), undo / cherry-pick (history browser), create / delete / move - the bind runs `custom.invalidate_git`, which drops the status, log and author caches so the columns and panels refresh immediately on the next render. Navigation does not invalidate; a real change does.

This is a caching strategy, not background loading, and deliberately so: xplr 1.1.0 cannot fill panels in asynchronously. A subprocess can only message xplr (`$XPLR_PIPE_MSG_IN`) while it is the running foreground bind - the pipe is torn down the moment the bind returns, so a detached "compute in the background, then refresh" job cannot signal xplr - and xplr does not redraw until a bind completes, so it cannot paint the listing first and fill git in later. The render functions therefore stay synchronous; the win is not recomputing what has not changed.

### Git context across symlinks

`repo-root.sh` works out which repo a directory belongs to. A bare `git -C <dir> rev-parse --show-toplevel` chdir()s into `<dir>`, which resolves symlinks physically - so entering a symlinked folder would show the git history of wherever the symlink points, not the repo you were browsing. Instead the helper walks the directory's _logical_ path, finds the deepest component that is itself a symlink, and discovers the repo from that symlink's real parent - the directory the symlink lives in - so a symlinked folder (and anything below it) keeps the git context of where you entered it from. Symlinks _above_ the anchor (say a symlinked `$HOME`) still resolve normally. If the anchor is in no repo - you reached a repo _by_ following a symlink from a non-repo directory, like `~/.config/xpdt` pointing at the checkout - it falls back to the target's own repo, so that case still shows git. `init.lua` (the panels, cached per directory in `repo_root_cache`) and every git browser go through this helper, so they always agree on the repo. Inside a symlinked directory the modified and author columns stay blank, since the origin repo does not track the symlink's contents.

### Git history: local vs pushed commits

`render_git_graph` marks each commit in the history box by whether it is on the remote. It computes the unpushed set once (per `GIT_LOG_TTL` cache refresh) with `git rev-list @{u}..HEAD` - the commits ahead of the branch's upstream - and logs with `git log --format="%H%x09%s%x09%an"` so it has each full sha to test against that set. Commits in the set get a hollow yellow dot (`○`, local / not pushed); the rest get the plain filled dot (`●`, on the upstream). If the branch has no upstream (`@{u}` fails), the set stays empty and every commit keeps the filled dot. This is the per-commit companion to the `↑ahead ↓behind` counts already shown in the panel title. The `;` commit history browser marks its rows the same way - see Commit history below.

### .xplrignore

`custom.apply_xplrignore` (run on load and on directory change) reads a `.xplrignore` file in the current directory. Lines beginning with `!` are kept (whitelist), other lines are hidden. This is used to show only the relevant top level folders when xplr is opened at a directory that contains several repos.

### Switching between git repos (`w`)

When a directory holds several git repos side by side (say `work/project1` and `work/project2`), `w` hops straight to the top of the next one. `next-git-repo.sh` works out the "workspace": if you are inside a repo it is that repo's parent, otherwise it is the nearest ancestor of the current directory that has git-repo children. It lists that workspace's immediate subdirectories that are git repos (sorted), finds where the current repo sits, and prints the next one (wrapping); from outside any of them it prints the first. The keybinding feeds that path to `xplr -m 'ChangeDirectory'`, so each press lands you at the top of a repo; the git panels then render from cache or, for a repo not seen yet, compute once.

### Changes browser (`enter`)

`git-changes-browser.sh` lists staged and unstaged entries and binds single keys (lazygit style, so the list is `--disabled` and `--no-input`, no filter box at all since the keys are actions not text): `s` toggles the entry (stages an unstaged entry with `git add`, unstages a staged entry with `git restore --staged`, based on the entry's group), `d` discard, `c` commit. Each action runs then `reload`s the list in place. `r` re-runs the list command on demand, so a change made outside xpdt (an edit or commit in another terminal) shows up without leaving the browser. The visible list is sized to the change count: a `load:transform` bind (which re-fires on every reload, including `r`) reads `$FZF_TOTAL_COUNT` and sets the preview window to `TERMH - min(count, 20) - OVER`, where `OVER` is the wrapped header's line count plus the preview's two border rows. Getting `OVER` right is the fix for the list collapsing to a row or two on a narrow terminal: the header wraps to several lines there, but the old code budgeted a flat three rows for chrome, so the oversized preview swallowed the list. `→` is context-aware - on an **unstaged** entry it opens the working file in Neovim to edit (`cd $ROOT && nvim`), on a **staged** entry it opens the read-only inline diff viewer (a staged snapshot is not something you edit in place; unstage it with `s` first to edit it); `enter` does nothing (opening is `→` only, across the whole app). `→` `reload`s the list afterwards since an edit may change the diff. Stage/unstage, discard and commit are separate scripts (`git-stage.sh`, `git-discard.sh`, `git-commit.sh`) so each can run behind the confirmation gate; the stage bind is therefore `execute` (it may need to prompt for the code) rather than the old `execute-silent`. The browser no longer exits when there are no changes: it opens on an empty list (fzf is fed truly empty input, not a blank row), every action is guarded on a non-empty focus so the keys are harmless no-ops, and `r` still refreshes - so you can open it, wait, and watch changes land. (fzf runs neither the preview nor the `load` event on an empty list, so the empty state is just the header over an empty pane, the same as the stash browser.)

### Partial staging (hunks) (`p`)

`p` on a focused entry opens `git-hunk-browser.sh`, a `git add -p` style view of that one file's hunks. `git-hunk.sh` is the engine, with three modes over the file's live diff (`git diff` for an unstaged entry, `git diff --cached` for a staged one):

- `list` prints one row per hunk (`N  @@ ...`), the index `N` first so the browser keys off it.
- `show` prints just the Nth hunk for the preview, piped through `bat --language=diff`.
- `apply` rebuilds a one-hunk patch - the file header (the lines before the first `@@`) plus the Nth `@@` block, extracted from the diff with `awk` - and pipes it to `git apply --cached` (adding `--reverse` when the entry is staged, to unstage). It reads the diff live each time, so after a hunk is staged the list `reload`s and the remaining hunks are re-extracted against the new index; sequential hunk staging stays valid even as line numbers shift. A new/untracked file has no diff, so the browser says so and points you at `s` (whole-file stage). `discard` is the same extraction piped to `git apply --reverse` against the working tree (no `--cached`), so it reverts just that hunk; it is offered for unstaged hunks only. Hunk apply goes through the confirmation gate under its own `hunk` action, hunk discard under the `discard` action.

### Commit history (`;`) and undo

`git-log-browser.sh` has two levels: the commit list, then the files changed in a commit. `ctrl-z` on the commit list runs `git-undo.sh`, which does `git reset --soft HEAD~1` (the commit is removed but the changes stay staged, matching VS Code's "undo last commit") after a confirm.

Each commit row carries a local-vs-pushed dot (after the hash), the same convention as the history box: `git-log-list.sh` builds the "local" set with `git rev-list <viewed-ref> --not --remotes` (commits reachable from the ref but not on any remote-tracking branch), so a yellow ○ means local / not pushed and a ● means pushed. Because it keys off the ref being viewed, pointing the browser at a remote branch with `b` shows all ● (everything is on the remote), and a repo with no remotes shows all ● (nothing to compare against, like the box's no-upstream case). The dot is field 2; field 1 stays the bare short hash so the `{1}` preview, cherry-pick and undo binds are untouched.

The browser can view any branch's history, not just the current one. `b` runs `git-branch-pick.sh` (fzf over local + remote branches) and writes the choice to a temp state file; the list command is `git-log-view.sh`, a thin adapter that reads that file (empty = current HEAD) so the fzf `reload` has no parentheses of its own. The header (`git-log-header.sh`, refreshed with `transform-header`) shows the viewed branch and the current branch. `ctrl-p` runs `git-cherry-pick.sh` on the focused commit: it refuses a dirty tree, goes through the `cherry-pick` gate, then `git cherry-pick <hash>` onto the current branch; on a non-zero exit (conflict) it runs `git cherry-pick --abort` and reports, so the branch is never left mid-conflict. So the workflow is: `b` to point at the source branch, `ctrl-p` on the commit to bring it across.

### Stash browser (`s`)

`git-stash-browser.sh` is the same lazygit-style, `--disabled --no-input` fzf shell as the changes browser, over `git stash list` (via `git-stash-list.sh`, which emits `stash@{N}` as the first field so every action keys off it). The preview is `git stash show -p --color=never` piped through `diff-words.py` (word-level highlighting) for the focused stash. `a` (apply), `p` (pop), `d` (drop), `n` (new), and `x` (clear) each shell out to `git-stash-op.sh` and then `reload` the list in place; `enter` / `→` pipe the full stash diff into a pager. `git-stash-op.sh` holds the confirms and prompts: `push` refuses a clean tree, prompts for an optional message, and always stashes untracked files too (`--include-untracked`); `apply` / `pop` show git's output and pause only if it fails (a conflict); `drop` is a y/N confirm; `clear` uses the same 2-digit code confirm as the risky git-menu entries. Every action guards on a non-empty ref, so the browser is safe to open (and to press `n` in) even with no stashes.

### Confirmation gate (`,`)

Every mutating action is guarded by a "type two random digits to confirm" gate, on by default for all of them. It is one place - `gate.sh` - so the behaviour and the toggles are uniform:

- Each action script calls `sh gate.sh confirm <action> "<message>"` and only proceeds on exit 0. `confirm` looks the action up in the config: if it is not gated it exits 0 immediately (the action just runs); if it is gated it prints the message, generates a random two-digit code, flushes the tty (so buffered paste cannot auto-answer), and reads the reply - exit 0 only on an exact match, otherwise it prints `Cancelled.` and exits 1. If it cannot open the tty it fails closed (exit 1), so an action never runs unconfirmed.
- The actions covered: `create`, `move`, `delete` (main view); `stage`, `hunk`, `discard`, `commit` (changes browser); `undo`, `cherry-pick` (commit history); `stash-apply`, `stash-pop`, `stash-drop`, `stash-new`, `stash-clear` (stash browser); `checkout`, `pull` (git menu).
- State lives in `~/.config/xpdt/.gate-config` as `enabled=1/0` (the master switch) plus one `<action>=1/0` line each. An absent file or an absent key reads as **on**, so the gate is enabled by default - including immediately after install, before the file exists. The installer also seeds an explicit all-on file (without overwriting an existing one), and `gate.sh` writes the file the first time you toggle anything.
- `,` opens `gate-menu.sh`: an fzf list (lazygit style, `--disabled --no-input`) of the master switch and every action with an `[x]` / `[ ]` checkbox. `enter` / `space` / `→` toggle the focused row (via `gate-toggle.sh`, which runs `gate.sh toggle` then emits `reload-sync(...)+pos({n}+1)` so the menu rebuilds with the cursor kept on that row - a plain async `reload` would race the `pos` and jump to the top); `←` / `esc` / `q` close. The action key is field 1 of each row, hidden with `--with-nth=2..` and used only by the toggle bind. With the master switch off the action rows are shown dimmed.
- The list is split into three labelled sections, separated by blank lines so the groups read as clearly distinct: **General** (the non-gate display settings), **Neovim intellisense** (per-language LSP toggles - see below), and **Confirmation gate** (the master switch, then the per-action toggles under an "Actions guarded by the code:" sub-header). The section headers and the blank spacer rows are ordinary rows carrying the key `#h`, which `gate-toggle.sh` and `gate.sh toggle` both ignore - landing on one and activating it is a harmless no-op that just keeps the cursor where it is.

Turning a gate off means that action runs on a single key with no prompt, so the master switch off is genuinely "no confirmations anywhere" - the default is deliberately the safe one.

The **General** section holds display settings that are not gates. One is **show hidden files** (dotfiles). It is stored in the same `.gate-config` (`show-hidden=1/0`, absent = on) and read at startup by `init.lua` (`read_bool_setting`) to set `general.show_hidden`. xplr 1.1.0 has no runtime message to change `show_hidden` (there is no `ToggleHidden`), so a toggle in the menu takes effect on the next launch rather than live. This replaces xplr's built-in `.` toggle, which xpdt now unbinds, so hidden-file visibility is a deliberate, persisted setting rather than an accidental keypress.

The other **General** setting is **claude-integration** (default off, so `gate.sh get` special-cases it): when on, the `claude` box below git history is a live panel of **all** your recent Claude Code sessions across every repo (not just the current one), grouped by repo with the current repo first. Each session shows a status glyph, its name, an approximate **context-usage %**, how long since it was last active, and - for a non-default model - the model; the session Claude is actively working in also shows its current task. A footer summarises the counts, e.g. `2 working · 1 waiting · 1 idle`.

Three states, from the newest user/assistant event in the log plus recency: **working now** (green `●`) - an unfinished turn written within ~90s; **waiting on you** (amber `◆`) - a finished turn (`stop_reason` `end_turn`/`stop_sequence`) within the last 30 min, i.e. Claude replied and is waiting for input; **idle / stale** (dim `○`) - older than that, or owed-but-gone-quiet. When two active sessions sit on the same repo and branch, the repo header is flagged (a clobber risk worth seeing).

Claude Code stores each session as `~/.claude/projects/<encoded-cwd>/<id>.jsonl`, appending one event per line. `claude-status.sh` scans the logs modified in the last few hours (excluding `subagents/`) and reads only each file's **tail** for: the name (`ai-title`), current task (`last-prompt`), branch, model, token usage (`input + cache_creation + cache_read` -> the context %, against a best-effort 200k/1M window), the newest turn (-> the status) and recency; a session's repo is the git top-level of its main `cwd`. Sessions sort working -> waiting -> idle and are capped with a `+N more`. There is no true liveness signal in the logs (closing Claude leaves no marker), so recency is the proxy. Reading just the tail keeps it cheap on multi-MB logs.

`init.lua` caches the result per repo (`CLAUDE_TTL`) and renders it in the `claude` box (`render_claude`); the box is given 0 height (hidden) whenever there is nothing to show, and the scan is skipped entirely when the setting is off (`read_bool_setting` guard, no spawn). It uses only the session files - no tmux, no process inspection - so it works on a plain single-machine install.

### Neovim intellisense (LSP)

The **Neovim intellisense** section of the `,` menu is a list of languages and frameworks (Lua, Python, Django, TypeScript / JavaScript, HTML, CSS, JSON, Bash, Rust, Go, Tailwind, Svelte, ESLint) you can turn on individually. Django is the template language server (`djlsp` from `django-template-lsp`: completion for template tags, filters, `{% url %}` names, `{% static %}` paths, block names and context variables); Python code itself is covered by the Python entry (pyright). Those _project-specific_ Django completions need djlsp to import your project, which needs its virtualenv - and djlsp only searches `env` / `.env` / `venv` / `.venv` under the project root and ignores an active venv, so it otherwise falls back to generic completions ("Failed to collect project-specific Django data"). xpdt therefore points djlsp at `$VIRTUAL_ENV` first (its `env_directories` init option, `django_env_directories` in `nvim/init.lua`), so an activated venv - from `venv`, poetry, pipenv or conda, all of which export `$VIRTUAL_ENV` - is picked up, with the in-project dirs as fallback. Launch xpdt from an activated venv (or keep the venv in one of those dirs) and it collects your real tags / urls / context. Each is stored in `.gate-config` as `lsp-<name>=1/0` (all default **off**). Ticking one makes Neovim **install that server for you** and start it - and only the languages you tick are ever fetched, so nothing heavy is pulled in by default.

The wiring lives in the Neovim config (`nvim/init.lua`), which xpdt hands off to (`→`). It uses Neovim's built-in LSP client (`vim.lsp.enable`, 0.11+) and built-in completion (`vim.lsp.completion`, no completion plugin), with **nvim-lspconfig** supplying each server's config - the correct `cmd` / root / `init_options` / `settings`, including the fiddly servers a hand-rolled config gets wrong (`eslint` needs a `settings.workspaceFolder`, and TypeScript uses `vtsls`, which bundles its own TypeScript, rather than `typescript-language-server`, which fails to start without a project `typescript`) - and **Mason** (`mason.nvim`, the standard server manager) installing the servers:

- On startup it reads `~/.config/xpdt/.gate-config`. For every enabled `lsp-<name>` whose server binary is already on `PATH` it starts the server with `vim.lsp.enable`, which reads lspconfig's config for it (a small `SERVERS` table maps each language to its lspconfig server name - `lua_ls`, `vtsls`, `eslint`, ... - and Mason package; we layer only small overrides, e.g. the `vim` global for Lua). For an enabled server that is **not** yet installed, it asks Mason to install that server's package; Mason fetches it into its own isolated dir - which it adds to `PATH`, so no global `npm -g` / `pip` installs - and xpdt starts it the moment the install finishes, re-attaching any open buffer. The first time you enable a language the server has to download, so it is ready a moment later (or on the next launch); after that it starts instantly.
- `:XpdtLsp` lists every language: `off`, `ON - installed`, or `ON - installing via Mason`. `:Mason` opens Mason's own UI to watch progress or manage servers by hand.
- When a server attaches, built-in completion is enabled (autotrigger, plus `Ctrl-Space`) and the usual LSP keys are bound - each only if the attached server supports that method, so an unsupported key (e.g. `gi` on a server with no implementation support) keeps its default Vim behaviour instead of warning: `K` hover, `gd` / `gr` / `gi` / `gD` navigation, `<leader>rn` rename, `<leader>ca` code action, `<leader>f` format, plus `<leader>e` and `]d` / `[d` for diagnostics (always on). Diagnostics show as wrapped virtual lines beneath the current line (the full message, on as many rows as it needs, rather than inline virtual text that ran off the right edge and could not wrap), with an underline and a gutter sign marking every diagnostic line; `<leader>e` opens the full text in a float. Neovim's own `virtual_lines` does not soft-wrap a long message, so the config word-wraps it in a `format` function to the width actually left for the text (window width minus the gutter and the column the message is indented under), which keeps even very long type errors on-screen. These are in the `ctrl-h` cheat sheet too.

`setup_lsp()` runs inline as `init.lua` finishes (not deferred) - `vim.lsp.enable` only auto-attaches buffers opened after it runs, and the file you launched on is read just after, so deferring would miss it; it force-loads Mason first so Mason's bin dir is on `PATH` by then. The language list is the same across `gate.sh`'s `lsp_rows` and the `SERVERS` + `MASON` tables in `nvim/init.lua` (all keyed by `lsp-<name>`), so adding a language is a small change in each.

### Inline diff viewer

`diff-view.sh` shows the whole file syntax highlighted by `bat` (the same look as the file preview), with removed lines (red) shown inline next to added lines (green) and context (grey), opened at the first change.

- Both the before and after versions of the file are written to temp files named after the real file (so `bat` detects the language) and syntax highlighted by `bat`.
- `diff-render.py` reads the `git diff -U0` hunks and interleaves the two highlighted versions: context and added lines come from the after version, removed lines from the before version, each gutter coloured by type (grey / green / red). It records the display position of the start of each change block.
- Changed lines carry the same word-level highlighting as the diff previews: a removed line gets a dim red background, an added line a dim green one, and the exact words that differ (paired removed / added lines word-diffed with `difflib`, unless too dissimilar) get a brighter background. Here it is layered over bat's syntax colours - only the background is switched, never reset mid-line, so the foreground highlighting is kept and the tint survives the soft-wrap.
- `→` and `shift-→` are bound to `transform` actions that call `diff-nav.sh`, which reads the current line index (`{n}`), finds the previous / next change block, and echoes a `pos(N)` action back to fzf to move the cursor. Both wrap around. fzf runs with `--scroll-off=9999`, which keeps the focused line vertically centred (fzf cannot honour that many context lines on both sides, so it balances them), rather than letting a jumped-to change land at the bottom edge.

Before / after are `git diff --cached` / index and HEAD for staged, working file and index for unstaged, and the commit and its parent for a commit.

### Diff previews (word-level highlighting)

The fzf diff previews - the commit history browser (both the commit and the per-file preview), the changes browser, the stash browser and the hunk browser - pipe an uncoloured unified diff (`git show` / `git diff` / `git stash show` with `--color=never`) through `diff-words.py`, which does two things a plain diff does not:

- **whole-line tint**: a removed line gets a dim red background and an added line a dim green one, so the changed "sentence" reads as a block (the same look as the inline viewer), not just a coloured `+` / `-` in the margin;
- **word-level emphasis**: paired removed / added lines are word-diffed with `difflib`, and only the words that actually differ get a brighter, bolder background (bright red for removed text, bright green for new text). So replacing one word in a line shows the old line dim-red with that word bright-red and the new line dim-green with the replacement bright-green, instead of the whole line lighting up. Removed and added lines are paired positionally within a hunk; a pair that is too dissimilar (a delete that merely landed next to an unrelated insert) is left dim rather than word-diffed, so unrelated lines do not light up.

The preview windows also set fzf's `wrap` flag, so long diff lines wrap onto the next row instead of being truncated at the pane edge. The filter never emits a full reset mid-line (it swaps only the background between the dim and bright tints), so the tint survives the soft-wrap. Tabs are expanded to four spaces so the background fills cleanly.

### File and content search (`/` `\`)

`search.sh` is the backend. `/` (files) uses `find` piped through `stat` and `sort` so results are newest modified first, and fzf runs with `--exact` so matches are literal substrings, not fuzzy; it also re-runs on every keystroke (fzf `change:reload`), so files created while the menu is open show up without leaving it. `\` (content) uses `ripgrep --fixed-strings --sortr=modified` (falls back to `grep`), re run on every keystroke via fzf's `change:reload`. Both emit each hit as a path relative to the scope dir (leading `/`) rather than the full launch path; `resolve.sh` turns a shown path back into the real one, which `→` then acts on (only `→`, never `enter`): a file opens in Neovim, and a folder hit (in `/`) is entered. `←` is `transform:[ -n {q} ] && backward-delete-char || abort`, so holding it deletes back through the query and, at the empty query, exits the search (fzf exposes no query-cursor position, so it deletes rather than moving a cursor). That exit touches `/tmp/xpdt-left-exit` so the post-search `flush-input.sh` drains a held key's auto-repeat until release rather than overshooting into xpdt - the same flag and drain the Neovim left-exit uses (Neovim knows its cursor, so it walks the cursor back and exits at the line start instead of deleting).

Scope: `scope.sh` reads / toggles `.search-scope` (`here` = current directory, `root` = the directory xplr was launched from). `tab` toggles it, and because the state is a file it persists across sessions and is shared by `/` and `\`. The current scope is shown in the fzf header.

### Neovim settings

The `,` menu's **NEOVIM** section has four toggles, all off by default:

- **Preview file before opening in Neovim** - `→` on a file shows the full-screen preview first (below) instead of opening Neovim straight away; from the preview, `→` / `ctrl-e` opens Neovim at the line. `open-or-preview.sh` reads the setting live (per keypress) and dispatches to `preview-file.sh` or `open-file.sh`, so the main view and both searches behave the same.
- **Left at the start of a line exits Neovim** - in Neovim (opened from xpdt), with no unsaved edits in any buffer, `←` at column 1 quits back to xpdt, so `left` keeps meaning "back" once the cursor can go no further left. It never fires when a buffer is modified, and rebinds only the arrow, not `h`. Read from `.gate-config` when Neovim starts. When you _hold_ `←` to trigger it, xpdt stays put and drains the key's auto-repeat until you release it (`flush-input.sh`), then resumes - so holding no longer walks you back up through directories once the editor closes (it reappears when you let go). A one-shot flush would miss this, because auto-repeat only starts after an initial delay - i.e. after Neovim has already closed.
- **Key-hint bar at the top of the window** - when on, Neovim shows a one-line help bar (its `winbar`) across the top of the window with the keybinds you are likely to forget (`spc Y` copy file, the LSP keys, `gc` comment, leap/surround, `:w`/`:q`, ... and `ctrl-h` for the full cheat sheet). winbar is a single line and cannot wrap, so the hints are width-packed - the bar shows as many as fit the window, most-useful first - and it advertises `←` back only when the left-exits setting is also on. Keys are bright, labels dim, both linked to the active colour theme. Read from `.gate-config` when Neovim starts, so it applies to the next file you open.
- **Show unstaged changes inline in Neovim** - in the changes browser, `→` on an **unstaged** entry normally opens the file in Neovim to edit. With this on, it opens the file with its unstaged changes shown **inline** (`:XpdtDiff`) - you edit the real file in one window (no split): added / changed lines carry a `+` / `~` sign and a subtle line tint, and the removed lines appear inline as red virtual lines where they were. It is a `vim.diff` against the git index, recomputed live (debounced) as you edit, so the marks track your edits; run `:XpdtDiff` again to toggle it off. It opens on the first change (centred), and `]c` / `[c` jump to the next / previous change (wrapping, with a `hunk N/M` readout), so the other diffs are easy to reach. Staged entries are unchanged (still the read-only inline diff viewer); `:XpdtDiff` also works by hand in any tracked file. Colours are the theme's `DiffAdd` / `DiffChange` / `DiffDelete` groups. (A removed block above the very first line can't render at the top edge - a Neovim limitation - so a change on line 1 shows the marker but not the old text; everywhere else shows it.)

### File preview

The full-screen file preview is shown by `→` (in the main view and on a `/` `\` search hit) when the "Preview file before opening in Neovim" setting is on - otherwise `→` opens Neovim directly (the `open-or-preview.sh` dispatch). In the preview, `→` (or `ctrl-e`) opens Neovim at the line you are on and `←` goes back; `enter` does nothing. `preview-file.sh` pipes `bat` (colour, tabs expanded) into `wrap-lines.py`, then into fzf. `wrap-lines.py` is the interesting part: it adds its own line number gutter and wraps long lines so the continuation lines line up under where the original line's content started (indent aware soft wrap), all while preserving `bat`'s ANSI colours across the wrap. It also remaps the target line for the `\` search jump, because wrapping turns one logical line into several display rows. `ctrl-y` copies the file with `pbcopy` and briefly flips the prompt to "copied to clipboard", auto reverting after two seconds via fzf's `--listen` HTTP port. fzf resets the cursor to the top whenever the query changes, so clearing a search would otherwise jump you away from the line you found; `preview-track.sh` (bound to the `result` event, which fires after filtering settles so `{n}` is the real focused line) records the landing line while you type and emits a `pos(...)` to restore it when the query goes empty.

### Colour themes

A `theme` setting in the `,` menu (radio: `theme=<name>` in `.gate-config`, default `monokai`) recolours the whole setup at once - xpdt, Neovim, the bat previews / diffs and the fzf browsers - across five themes: Monokai, Gruvbox, Nord, Dracula and Tokyo Night. xplr and Neovim read the setting only at startup, so a theme change takes effect on the next `xpdt` launch.

- **xpdt** (`theme.lua`): the base node types (directory / file / symlink) and the dimmed / special groups (`.git`, `node_modules`, ...) are recoloured from a per-theme palette at load, keeping the icon glyphs and the semantic per-extension colours (a Python file stays Python-yellow on every theme).
- **Neovim** (`nvim/init.lua`): applies the matching colorscheme (`monokai` / `gruvbox` / `nord` / `dracula` / `tokyonight-night`; the plugins are bundled). Only Monokai layers the hand-tuned bat-matched palette - the others' own colorschemes already match their bat theme.
- **bat** (previews and diffs) and **fzf** (browsers / popups): `theme-env.sh`, sourced by the `xpdt` launcher, maps the setting to `BAT_THEME` and an `FZF_DEFAULT_OPTS --color` set, exported so every child inherits them. Gruvbox / Nord / Dracula / Monokai are built into bat; the Tokyo Night bat theme is fetched and built into bat's cache by `install.sh` (pinned to a `folke/tokyonight` release).

`theme.lua` also sets the node-type icons: a folder icon for directories, a link icon for symlinks, and per-extension icons for common languages. Icons are Nerd Font glyphs, so the terminal font must be a Nerd Font (Hack Nerd Font and MesloLGS NF are known to work) or they render as boxes.

### Help popups (`h`, `ctrl-h`)

The key legend used to be an always-on `CustomList` panel at the bottom of the screen, but its packed rows ran off the side of narrow terminals. It is now on demand: `h` runs `help.sh` and `ctrl-h` runs `nvim-cheatsheet.sh`, each of which prints coloured, sectioned, one-key-per-line text and pipes it into `popup.sh`. `popup.sh` is a small `fzf` wrapper (`--disabled --no-input`, a rounded border and a margin so it floats over the frame) that shows the text as a scrollable popup; arrows or the mouse wheel scroll, and `q` / `esc` / `left` (or the opening key again) close it. Because the layout no longer reserves those rows, the file listing and git history get the space back.

## Conventions and gotchas

- fzf is doing most of the interactive work. Patterns used: `reload` to refresh a list in place after an action, `execute` (takes over the terminal, for pagers / prompts) vs `execute-silent` (no screen switch), `transform` (its stdout is parsed as fzf actions, used with `pos(N)` for the diff navigation), the `[...]` delimiter instead of `(...)` when a bind's command itself contains parentheses, `--listen` for updating the UI from a background process (the copy toast), and `--disabled` when single letter keys should be actions rather than filter input.
- Pasted input can leak past a `read` prompt and be interpreted by fzf as key presses (which fired stray bindings). All confirm / commit prompts flush the terminal input queue with `termios.tcflush` before / after reading so buffered paste cannot auto answer them.
- fzf runs `execute()` binds with the tty still in raw mode and restores it inconsistently, so a `read` prompt inside one can see Enter as a bare carriage return (shown as `^M`) that never ends the line. `gate.sh confirm` runs `stty sane` before reading to force cooked mode; fzf re-applies its own mode when the bind returns.
- Those confirmation prompts are printed to `/dev/tty`, so they linger on the normal screen after the bind returns. The full-screen viewers launched from within an fzf session clear the screen (`\e[2J`) to remove them - but only **after** they have prepared their content, right before fzf paints, not before. `preview-file.sh` and `diff-view.sh` render into a temp file / variable first (the `bat` + wrap / diff-render step, which can take a moment on a big file) without touching the screen, so the current view stays put during preparation and the screen swaps to the ready view in one paint. Clearing first (the old order) left the screen blank for the whole prep - a visible flash to blank before the file or diff appeared.
- fzf and xplr hide the terminal cursor (`\e[?25l`) for their own UI and do not restore it for an `execute` / `BashExec` child, so a `read` prompt had no visible caret. The input prompts (the 2-digit code in `gate.sh`, commit messages in `git-commit.sh`, stash messages in `git-stash-op.sh`, file names in `file-op.sh`) emit `\e[?25h` (show cursor) before reading - folded into the same `\e[2J\e[H\e[?25h` they use to clear.
- fzf truncates a long `--header` rather than wrapping it, which cut off the key legend on a narrow terminal. Every browser now runs its header through `wrap-header.sh`, which greedily packs the space-separated items into lines that fit the tty width and passes the result back as a multi-line header (fzf renders one header line per `\n`); items are never split across a break.
- macOS captures `ctrl-left` / `ctrl-right` for Mission Control ("move a space"), so they never reach the terminal. That is why the diff viewer's previous / next uses `→` and `shift-→` rather than a ctrl arrow.
- Icons need a Nerd Font selected in the terminal (see Theme).
- Caches (see Git modified column and status above for the strategy): `git_status_cache` and `git_log_cache` on a ~10s TTL backstop (`STATUS_TTL` / `GIT_LOG_TTL`), `git_author_cache` / `git_author_dir_done` / `repo_root_cache` for the session, and `CLAUDE_TTL` (~5s) for the claude panel. All of them (except `repo_root_cache`) are dropped immediately by `custom.invalidate_git` after any xpdt action that changes git state - stage / commit / discard / stash / checkout / pull / undo / cherry-pick / create / delete / move - so the columns and panels refresh on the next render; a change made outside xpdt is caught within the TTL. Navigation does not invalidate.
- Helpers are plain POSIX `sh` except `git-commit.sh` (bash, for `read -e` arrow key line editing) and the `.py` helpers (Python). `git-commit.sh` must be invoked as `bash ...`, not `sh ...`, or `read -e` is lost.
- Python startup is the app's dominant per-interaction cost: a bare `python3` spawn is ~16ms (vs ~1ms for `sh`/`awk`/`git`), most of it the `site` import. So (1) work that a shell can do is not handed to python - `wrap-header.sh` is `awk`, the confirm code in `gate.sh` is `/dev/urandom` via `od`; and (2) every genuine `.py` helper and `python3 -c` one-liner is run with `python3 -S` (skip `site`), which roughly halves startup to ~7ms. `diff-words.py` runs on every browser preview move and `wrap-header.sh` on every menu open, so these paths are the ones worth keeping python-free or `-S`. git itself is not the bottleneck (status/log/author are cached and a few ms each - see the caching notes above).

## Testing

There is no automated test harness. Changes were verified with `luac -p init.lua` / `theme.lua` (Lua parse), `sh -n` / `bash -n` on the shell helpers, functional tests of the git logic in throwaway repos under `/tmp`, and driving xplr headlessly in a pseudo terminal (Python `pty`) to confirm rendering, colours, icons and that the config loads without error. The live behaviour of interactive fzf key handling can only be confirmed by using it.
